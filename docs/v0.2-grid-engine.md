# v0.2 — Grid engine

**Goal:** A reusable, game-agnostic engine that powers all three games. This is the
heart of the project: a fixed-timestep loop on a discrete grid, with collision,
buffered input, pause, and score/state plumbing. Built and tested in isolation
before any specific game.

## Prerequisite reading

- `node_modules/next/dist/docs/01-app/` — client components / `"use client"`,
  hooks usage, and how effects behave under React 19 in Next 16 (the loop runs in a
  client component with `requestAnimationFrame`).

## Design

### Coordinates

```ts
type Position = { x: number; y: number };
```

All game logic operates on grid cells, never pixels. The renderer is the only code
that multiplies by the 16px cell size.

### Fixed timestep loop — `lib/engine/gameLoop.ts`

- Runs at **10–15 FPS** (one logical "tick" every ~66–100ms), independent of
  `requestAnimationFrame`'s 60Hz. Accumulate elapsed time; step logic in fixed
  increments; render the latest grid each frame.
- No interpolation between ticks — objects must snap cell-to-cell (LCD behavior).
- Exposes `start`, `stop`, `pause`, `resume`, and a per-tick `update(dt)` callback.

### Grid — `lib/engine/grid.ts`

- Define the canonical playfield (default **10 cols × 20 rows**, configurable per
  game) and helpers: bounds checks, cell get/set, clear, iterate.
- Canvas LCD renderer: draws each occupied cell as a `--lcd-dark` square with a
  `--lcd-outline` border on the `--lcd-background` field. Optional faint "off"
  segment dots for authenticity. No anti-aliasing.

### Collision — `lib/engine/collision.ts`

- Pure cell-overlap checks: point-vs-point, rect-vs-rect (for multi-cell objects
  like a 2×2 tank), and "is this cell occupied / out of bounds / a wall".

### Input — `lib/engine/input.ts`

- Keyboard buffering so a keypress between ticks isn't lost: Arrows = move,
  Space = fire, Enter = start, P = pause, M = mute.
- Debounce/repeat policy suited to discrete movement (hold-to-repeat at tick rate).

### State store — Zustand

- Shared slice: `status` (idle/playing/paused/gameover), `score`, `hiScore`,
  `level`, `lives`, `mode`, plus actions. High score read/written to `localStorage`.
- Difficulty levels affect tick rate and spawn rates (consumed by games).

## Deliverables

- `lib/engine/{gameLoop,grid,collision,input}.ts`.
- A Zustand store for shared game state + `localStorage` high-score persistence.
- A throwaway demo (a single blinking/moving block on the grid) proving the loop,
  rendering, input, and pause all work end-to-end.

## Out of scope

- Any real game rules, enemies, the handheld shell, audio.

## Acceptance criteria

- [ ] Loop ticks at a fixed 10–15 FPS regardless of monitor refresh.
- [ ] Movement is strictly cell-to-cell; zero interpolation/easing.
- [ ] Pause/resume works; input buffered correctly.
- [ ] Renderer uses only the 4 LCD colors.
- [ ] High score survives a page reload.
- [ ] Engine has no import dependency on any specific game.
